# Отчёт о проверке проекта перед сдачей

Все проверки выполнялись в изолированных Python-виртуальных окружениях
(отдельно для backend, Telegram-бота и VK-бота — см. `requirements/*.txt`)
и с реальными зависимостями (FastAPI, SQLAlchemy, Alembic, aiogram 3,
vkbottle), а также сборкой React-фронтенда через Vite.

## 1. Синтаксис и целостность файлов

Все 107 `.py`-файлов проекта пройдены через `ast.parse()` — ошибок нет.
В процессе разработки было обнаружено и устранено повреждение нескольких
файлов (обрыв записи / null-байты) при редактировании через инструмент
правки на смонтированной файловой системе — после обнаружения все такие
файлы были перезаписаны целиком и повторно провалидированы; финальный
прогон подтверждает 0 повреждённых файлов.

## 2. Импорты и циклические зависимости

- `app/*` + `admin_api/*` (79 модулей) — каждый модуль импортирован
  индивидуально в чистом окружении: 0 ошибок `ImportError`/`ModuleNotFoundError`,
  циклических импортов не обнаружено.
- `bots/telegram/*` (12 модулей, окружение с `aiogram==3.13.1`) — 0 ошибок.
- `bots/vk/*` (11 модулей, окружение с `vkbottle==4.4.4`) — 0 ошибок.
- Подтверждено, что `aiogram` и `vkbottle` **не** устанавливаются в одно
  окружение (несовместимые версии `aiohttp`) — поэтому у них разные
  `requirements/*.txt` и разные Docker-образы.

## 3. Обработчики и роутеры

- Telegram-бот: 5 роутеров (`start`, `purchase`, `payment_check`, `tickets`,
  `help`) успешно регистрируются в `Dispatcher`.
- VK-бот: 5 модулей-обработчиков (`start`, `purchase`, `payment_check`,
  `tickets`, `help`) успешно регистрируются в `Bot` через `register(bot)`.
- FastAPI: все 14 роутеров подключены в `admin_api/main.py`, подтверждено
  интроспекцией `app.routes` — 50 эндпоинтов `/api/*` (dashboard, participants,
  sales, manual-registrations, tickets, giveaways, settings, panel-users,
  broadcasts, reports, audit-log, auth, webhooks × 2 банка, health).

## 4. Модели SQLAlchemy и миграции Alembic

- `alembic upgrade head` и `alembic downgrade base` — пройдены на чистой
  SQLite базе без ошибок (полный roundtrip проверен дважды).
- `alembic check` после применения всех миграций: **"No new upgrade
  operations detected"** — модели и миграции полностью синхронизированы.
- Все внешние ключи, индексы и enum-поля создаются корректно.

## 5. Сквозные функциональные сценарии (через HTTP, ASGITransport)

1. Логин супер-админа → получение JWT → `GET /api/auth/me`.
2. Создание розыгрыша → открытие регистрации → попытка изменить
   `ticket_price` после открытия → корректный `409 Conflict`
   (`GiveawayImmutableError`).
3. Ручная регистрация: создание (`PENDING`) → выдача номерков (`CONFIRMED`)
   → номерки реально созданы и видны в `/api/tickets`.
4. Dashboard корректно агрегирует участников/номерки/выручку.
5. Журнал аудита фиксирует все перечисленные действия.
6. **Онлайн-оплата (с подменённым HTTP-вызовом в банк)**: создание платежа →
   webhook Т-Банка с корректной подписью (`CONFIRMED`) → статус `succeeded`,
   номерки выданы. Повторный (дублирующий) webhook с тем же payload —
   **идемпотентно проигнорирован**, повторной выдачи номерков не произошло
   (было выдано ровно 2 номерка, а не 4). Это была найдена и исправлена одна
   реальная ошибка в ходе проверки: `SettingsService.get_active_payment_provider`
   возвращал значение из `.env` как обычную строку вместо `PaymentProviderEnum`,
   что ломало создание платежа — исправлено явным приведением типа.
7. Webhook на неизвестный `order_id` — обрабатывается корректно (200 OK,
   запись в аудит, без исключений).

## 6. Совместимость с .env

`.env.example` успешно загружается через `pydantic-settings` в модель
`Settings` без ошибок валидации; все поля, используемые в коде
(`PAYMENT_PROVIDER`, `DATABASE_URL`, `TELEGRAM_BOT_TOKEN`, `VK_GROUP_TOKEN`
и т.д.), присутствуют в `.env.example`.

## 7. Frontend

- `tsc -b` (проверка типов TypeScript) — 0 ошибок.
- `vite build` — успешная сборка продакшен-бандла.

## 8. Docker Compose

- `docker-compose.yml` — валидный YAML; 4 сервиса (`backend`, `telegram-bot`,
  `vk-bot`, `admin-frontend`) + volume `sqlite_data`.
- Все пути `COPY` в четырёх Dockerfile'ах (`docker/*.Dockerfile`) проверены
  на существование в репозитории.
- Установка зависимостей "с нуля" в чистом виртуальном окружении (эквивалент
  шага `pip install` внутри Docker-образа) выполнена и проверена отдельно для
  backend/telegram/vk наборов зависимостей.

## Ограничение проверки

В среде, где готовился проект, отсутствовал доступ к Docker Engine, поэтому
итоговая команда `docker compose up -d --build` не была выполнена буквально.
Вместо этого каждый её отдельный шаг воспроизведён и проверен нативно:
установка зависимостей с нуля, миграции, полный HTTP-сценарий на реальном
ASGI-приложении, сборка фронтенда. Рекомендуется выполнить
`docker compose up -d --build` при первом развёртывании на машине с Docker
и свериться с логами (`docker compose logs -f`).
