# Отчёт о проверке проекта

Все проверки выполнялись в изолированных Python-виртуальных окружениях
(отдельно для backend и Telegram-бота — см. `requirements/*.txt`) и с
реальными зависимостями (FastAPI, SQLAlchemy, aiogram 3), а также сборкой
React-фронтенда через Vite.

> Архитектура менялась по ходу проекта: изначально были и Alembic-миграции
> (с опцией перехода на PostgreSQL), и VK-бот. Оба решения впоследствии были
> осознанно убраны (см. README, разделы «База данных» и «Известные
> ограничения») — проект работает только на SQLite (`create_all()`, без
> миграционного инструмента), участники — только через Telegram. Этот отчёт
> описывает текущее (актуальное) состояние.

## 1. Синтаксис и целостность файлов

Все `.py`-файлы проекта пройдены через `ast.parse()` — ошибок нет. В
процессе разработки было обнаружено и устранено повреждение нескольких
файлов (null-байты) при смешивании инструмента точечной правки с
bash-heredoc на одном и том же файле — после обнаружения такие файлы
перезаписывались целиком и повторно валидировались; финальный прогон
подтверждает 0 повреждённых файлов.

## 2. Импорты и циклические зависимости

- `app/*` + `admin_api/*` — каждый модуль импортирован индивидуально в
  чистом окружении: 0 ошибок `ImportError`/`ModuleNotFoundError`,
  циклических импортов не обнаружено.
- `bots/telegram/*` (окружение с `aiogram==3.13.1`) — 0 ошибок.

## 3. Обработчики и роутеры

- Telegram-бот: 5 роутеров (`start`, `purchase`, `payment_check`, `tickets`,
  `help`) успешно регистрируются в `Dispatcher`.
- FastAPI: все роутеры подключены в `admin_api/main.py`, подтверждено
  интроспекцией `app.routes` — 55 маршрутов `/api/*` + `/metrics`
  (dashboard, participants, sales, manual-registrations, tickets,
  giveaways, settings, panel-users, broadcasts, reports, audit-log, auth,
  webhooks × 2 банка, health, metrics).

## 4. Модели SQLAlchemy и схема БД

- `python -m scripts.init_db` (`Base.metadata.create_all()`) — пройден на
  чистой SQLite базе без ошибок; повторный запуск идемпотентен (ничего не
  падает и не дублируется).
- Все внешние ключи, индексы и enum-поля создаются корректно.
- Alembic и PostgreSQL в проекте больше не используются (осознанное
  архитектурное упрощение) — см. README.

## 5. Сквозные функциональные сценарии (через HTTP, ASGITransport)

1. `create_all()` → создание Super Admin (`scripts.create_superadmin`) →
   логин → получение JWT → `GET /api/auth/me`.
2. Создание розыгрыша → открытие регистрации → попытка изменить
   `ticket_price` после открытия → корректный `409 Conflict`
   (`GiveawayImmutableError`).
3. Ручная регистрация: создание (`PENDING`) → выдача номерков (`CONFIRMED`)
   → номерки реально созданы и видны в `/api/tickets`.
4. Dashboard корректно агрегирует участников/номерки/выручку.
5. Журнал аудита фиксирует все перечисленные действия.
6. **Онлайн-оплата (с подменённым HTTP-вызовом в банк)**: создание платежа →
   webhook Т-Банка с корректной подписью (`CONFIRMED`) → статус `succeeded`,
   номерки выданы. Повторный (дублирующий) webhook с тем же payload —
   **идемпотентно проигнорирован**, повторной выдачи номерков не произошло.
   Это была найдена и исправлена одна реальная ошибка в ходе проверки:
   `SettingsService.get_active_payment_provider` возвращал значение из
   `.env` как обычную строку вместо `PaymentProviderEnum`, что ломало
   создание платежа — исправлено явным приведением типа.
7. Webhook на неизвестный `order_id` — обрабатывается корректно (200 OK,
   запись в аудит, без исключений).
8. Автоматизированный тест-пакет (`pytest`, 31 тест, см. README раздел
   «Тестирование») покрывает эти же сценарии на изолированной in-memory
   SQLite — прогоняется в CI-независимом режиме одной командой.

## 6. Совместимость с .env

`.env.example` успешно загружается через `pydantic-settings` в модель
`Settings` без ошибок валидации; все поля, используемые в коде
(`PAYMENT_PROVIDER`, `DATABASE_URL`, `TELEGRAM_BOT_TOKEN` и т.д.),
присутствуют в `.env.example`.

## 7. Frontend

- `tsc -b` (проверка типов TypeScript) — 0 ошибок.
- `vite build` — успешная сборка продакшен-бандла.

## 8. Docker Compose

- `docker-compose.yml` — валидный YAML; сервисы `backend`, `telegram-bot`,
  `admin-frontend`, `caddy` + volume `sqlite_data`, `caddy_data`, `caddy_config`.
- Все пути `COPY` в Dockerfile'ах (`docker/*.Dockerfile`) проверены на
  существование в репозитории.
- Установка зависимостей "с нуля" в чистом виртуальном окружении (эквивалент
  шага `pip install` внутри Docker-образа) выполнена и проверена отдельно
  для backend/telegram наборов зависимостей.

## Ограничение проверки

В среде, где готовился проект, отсутствовал доступ к Docker Engine, поэтому
итоговая команда `docker compose up -d --build` не была выполнена буквально.
Вместо этого каждый её отдельный шаг воспроизведён и проверен нативно:
установка зависимостей с нуля, создание схемы БД, полный HTTP-сценарий на
реальном ASGI-приложении, сборка фронтенда. Рекомендуется выполнить
`docker compose up -d --build` при первом развёртывании на машине с Docker
и свериться с логами (`docker compose logs -f`).
