# Raffle Platform — платформа розыгрышей цифровых постеров

Production-ready платформа для продажи цифровых постеров и автоматической
регистрации участников розыгрыша. Участники взаимодействуют через
Telegram-бот, администрирование — через веб-панель.

> VK-интеграция полностью удалена из проекта (была реализована и работала,
> но сейчас не нужна в проде) — вернёмся к ней отдельно позже. Код можно
> восстановить из git-истории, если понадобится.

## Архитектура

```
                    ┌─────────────────┐
                    │      SQLite      │
                    └────────┬─────────┘
                             │
              ┌──────────────┴──────────────┐
              │                              │
      ┌───────▼───────┐            ┌─────────▼────────┐
      │ admin_api      │            │ bots/telegram    │
      │ (FastAPI REST) │            │ (aiogram 3)      │
      └───────┬────────┘            └──────────────────┘
              │
              │ REST API
      ┌───────▼────────┐
      │ admin-frontend │
      │ (React + TS)   │
      └────────────────┘
```

Ключевое архитектурное решение: `admin_api` и `bots/telegram` — это два
независимых процесса (два Docker-образа), но оба импортируют **один и тот
же** пакет `app/` (модели, репозитории, сервисы, платёжные провайдеры). Это
значит, что бизнес-логика (выдача номерков, финализация платежа, привязка
Telegram к участнику) существует в одном месте и работает одинаково
независимо от того, кто её вызвал — веб-панель, бот или webhook банка.

У backend'а и Telegram-бота разные зависимости (FastAPI vs aiogram), поэтому
у каждого — свой `requirements/*.txt` и свой Docker-образ.

Уведомления участнику после успешной оплаты (номерки + цифровой постер)
отправляются **напрямую** через HTTP Telegram Bot API из `admin_api` (см.
`app/services/notification_service.py`), а не из процесса бота. Это значит,
что подтверждение оплаты работает, даже если контейнер бота в этот момент не
запущен.

## Структура проекта

```
raffle_platform/
├── app/                    # общий пакет: модели, репозитории, сервисы, платежи
│   ├── models/             # SQLAlchemy 2.x модели
│   ├── repositories/       # Repository Pattern
│   ├── services/           # сервисный слой (бизнес-логика)
│   ├── payments/           # BasePaymentProvider + TBankProvider + VTBProvider
│   ├── integrations/       # прямые HTTP-вызовы Telegram Bot API
│   ├── schemas/            # Pydantic-схемы для admin_api
│   └── core/               # исключения, права доступа, логирование, security
├── admin_api/              # FastAPI: REST API веб-панели + webhook-эндпоинты
│   └── routers/
├── bots/
│   └── telegram/           # aiogram 3 — участники
├── admin-frontend/         # React + TypeScript + Tailwind — веб-панель
├── docker/                 # Dockerfile'ы, nginx.conf, Caddyfile
├── scripts/                # init_db.py, create_superadmin.py, backup_db.sh, auto_update.sh
├── docker-compose.yml
├── requirements/           # base.txt + backend.txt + telegram.txt + dev.txt
└── .env.example
```

## Быстрый старт

```bash
cp .env.example .env
# отредактируйте .env: TELEGRAM_BOT_TOKEN, ключи банков,
# APP_SECRET_KEY, INITIAL_SUPERADMIN_PASSWORD

docker compose up -d --build
```

Каждое следующее обновление — просто:

```bash
./deploy.sh
```

(git pull + пересборка и перезапуск Docker-сервисов; см. комментарии в
самом `deploy.sh`. Для автоматического запуска по расписанию вместо ручного
`./deploy.sh` — см. `scripts/auto_update.sh` и `docs/AUTO_UPDATE.md`.)

После запуска:
- Веб-панель: http://localhost:8080
- API + Swagger-документация: http://localhost:8000/docs
- Первый вход: логин/пароль из `INITIAL_SUPERADMIN_LOGIN` / `INITIAL_SUPERADMIN_PASSWORD`
  (создаётся автоматически при первом старте `backend`, см. `scripts/create_superadmin.py`)

Локальная разработка без Docker — см. `docs/DEVELOPMENT.md`.

## Переменные окружения (.env)

Полный список с комментариями — в `.env.example`. Ключевые группы:

- **APP_*** — секретный ключ, часовой пояс, публичный базовый URL (нужен для
  формирования notification_url платёжных провайдеров).
- **DATABASE_URL** — подключение к SQLite (единственная поддерживаемая БД,
  см. раздел «База данных»).
- **JWT_*** — параметры токенов веб-панели.
- **TELEGRAM_ENABLED**, **TELEGRAM_BOT_TOKEN**, **TELEGRAM_PROXY_URL**.
- **PAYMENT_PROVIDER** — `tbank` или `vtb`; можно переопределить в панели
  (раздел «Настройки») без передеплоя.
- **TBANK_*** / **VTB_*** — ключи эквайринга.
- **INITIAL_SUPERADMIN_LOGIN/PASSWORD** — создаются один раз при первом
  старте backend.
- **CORS_ORIGINS** — адрес(а) фронтенда.
- **DOMAIN**, **LETSENCRYPT_EMAIL** — reverse-proxy/TLS (Caddy).
- **BACKUP_DIR**, **BACKUP_RETENTION_DAYS** — резервное копирование БД.
- **AUTO_UPDATE_BRANCH**, **AUTO_UPDATE_NOTIFY_CHAT_ID** — автообновление.

## Бизнес-правила

- Главный идентификатор участника — **номер телефона**. Telegram ID вторичен
  и необязателен; привязывается автоматически при первом запуске бота, если
  участник с таким телефоном уже существует (`RegistrationService`).
- Количество покупок не ограничено; номерки всегда привязаны к телефону.
- Номерки выдаются **случайно и уникально** в пределах розыгрыша:
  `<PREFIX>-<000000>` (`TicketService`).
- После успешной онлайн-оплаты номерки выдаются автоматически, участник
  получает цифровой постер. При ручной продаже бот ничего не отправляет —
  постер физический.
- **Розыгрыши**: у каждого свой уникальный префикс; поддерживаются несколько
  независимых и одновременных розыгрышей. После открытия регистрации
  (`opened_at` заполнен) параметры `prefix`/`ticket_price`/`max_tickets`
  становятся неизменяемыми (`GiveawayImmutableError`). Блокировка
  (`is_locked`) запрещает выдачу новых номерков, но не мешает работе с уже
  выданными.
- **Ручные регистрации**: поток `PENDING → CONFIRMED` (выдача номерков) либо
  `PENDING → CANCELLED` (отмена возможна только до выдачи номерков).

## Платёжные провайдеры

Единый интерфейс — `app/payments/base.py::BasePaymentProvider`
(`create_payment`, `verify_and_parse_webhook`, `check_status`). Т-Банк и ВТБ
реализованы как отдельные классы (`app/payments/tbank.py`,
`app/payments/vtb.py`), выбор — переменной `PAYMENT_PROVIDER` в `.env` или
override в панели («Настройки»). Webhook каждого банка — отдельный роутер
(`admin_api/routers/webhooks_tbank.py`, `webhooks_vtb.py`).

Финализация платежа идемпотентна: строка платежа блокируется
(`SELECT ... FOR UPDATE`) на время перехода `PENDING → {SUCCEEDED|FAILED}`;
повторный webhook или резервная проверка после финализации — no-op
(`PaymentService._apply_terminal_status`).

**Добавление нового банка**: реализовать класс `BasePaymentProvider`,
зарегистрировать в `app/payments/factory.py`, добавить HTTP-роутер для его
webhook. Сервисный слой и боты не меняются.

> Контракт Т-Банка (Tinkoff Acquiring API v2) — публичный и задокументирован
> официально. Контракт ВТБ у разных мерчантов отличается и предоставляется
> банком при подключении — в `app/payments/vtb.py` реализован типовой паттерн
> (создание заказа / HMAC-подпись / webhook), поля/эндпоинты нужно сверить с
> реальным договором эквайринга ВТБ.

## Боты

Mobile-first, постоянное ReplyKeyboard-меню, inline-кнопки — только там, где
необходимо (выбор количества, способ оплаты, резервная проверка).

Функции: регистрация и подтверждение телефона, покупка номерков, выбор
розыгрыша (если их несколько), ссылка СБП/платёжной страницы, резервная
проверка оплаты, просмотр номерков, история покупок, помощь.

- **Telegram** (`bots/telegram`, aiogram 3): телефон подтверждается кнопкой
  «Отправить номер телефона» (`request_contact`).

### Временное выключение бота

`TELEGRAM_ENABLED=false` в `.env` — процесс бота запускается, но к Telegram
не подключается и ничего не делает (не крутит restart-loop). Это же
проверяется в backend: уведомления после оплаты (`NotificationService`) и
рассылки (`BroadcastService`) не пытаются слать через Telegram, если он
выключен, а создать рассылку при выключенном Telegram нельзя (явная ошибка
при создании — лучше сказать сразу, чем молча отправить 0 сообщений).

### Прокси для Telegram

Если сервер не имеет прямого доступа к `api.telegram.org`, задайте в `.env`:

```
TELEGRAM_PROXY_URL=http://user:password@proxy.example.com:8080
# или
TELEGRAM_PROXY_URL=socks5://user:password@proxy.example.com:1080
```

Поддерживаются HTTP(S) и SOCKS5, с авторизацией и без. Прокси применяется
одновременно в двух местах:

- в самом Telegram-боте (`bots/telegram/bot.py`) — через
  `AiohttpSession(proxy=...)`;
- в backend'е (`app/integrations/telegram_api.py`) — для прямых уведомлений
  после оплаты и рассылок, которые идут через HTTP, минуя aiogram.

Оставьте переменную пустой, если прямой доступ к Telegram есть.

## Автообновление сервера из Git

`scripts/auto_update.sh` — скрипт для сервера: по расписанию (cron/systemd
timer) проверяет `origin`, и если появились новые коммиты — подтягивает их
(`git pull --ff-only`) и пересобирает/перезапускает Docker-сервисы. Если
изменений нет — ничего не делает. Если в рабочей копии есть незакоммиченные
правки — останавливается, чтобы их не потерять. Подробности, настройка cron
и systemd-таймера — в `docs/AUTO_UPDATE.md`.

## Production-hardening

Набор эксплуатационных мер, применённых поверх базовой функциональности.

### Reverse-proxy и TLS

Сервис `caddy` (docker-compose) — единственная точка входа снаружи (порты
80/443). Terминирует HTTPS и проксирует запросы на `backend` (`/api/*`) и
`admin-frontend` (всё остальное). При заданном в `.env` реальном `DOMAIN` с
DNS, указывающим на сервер, сертификат Let's Encrypt получается и продлевается
автоматически, без ручных действий. Конфигурация — `docker/Caddyfile`.
`/metrics` бэкенда через Caddy не проксируется — доступен только внутри
docker-сети.

### Непривилегированные контейнеры

`backend`, `telegram-bot` — выделенный пользователь `app` (не root) внутри
образа, владеющий `/app`. `admin-frontend` — образ
`nginxinc/nginx-unprivileged`, который не требует root даже для старта
master-процесса (в отличие от обычного `nginx`, слушает порт 8080).

### Rate limiting

`/api/auth/login` — не более 10 попыток в минуту с одного IP (защита от
перебора пароля); `/api/webhooks/*` — не более 60 запросов в минуту с одного
IP (защита от флуда). Реализация — `app/core/rate_limit.py`, простой
fixed-window лимитер в памяти процесса (для деплоя из нескольких реплик
backend нужно перевести на общее хранилище, например Redis).

### Заголовки безопасности и CORS

`app/core/middleware.py:SecurityHeadersMiddleware` добавляет
`X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`,
`Permissions-Policy`, `Strict-Transport-Security`. CORS
(`admin_api/main.py`) сужен до реально используемых методов/заголовков,
`allow_credentials=False` (SPA авторизуется JWT в заголовке `Authorization`,
а не cookie).

### Request-id и структурированные логи

`app/core/middleware.py:RequestIDMiddleware` генерирует (или принимает от
Caddy) `X-Request-ID`, привязывает его к `structlog` contextvars — все
лог-записи одного запроса помечены одним `request_id`, что упрощает поиск
по логам при разборе инцидентов.

### Метрики Prometheus

`GET /metrics` (не проксируется наружу, см. выше) — латентность, количество
запросов и ошибок по каждому эндпоинту (`prometheus-fastapi-instrumentator`).

### Резервное копирование БД

`scripts/backup_db.sh` — консистентный дамп SQLite (`VACUUM INTO`), сжатие,
автоматическая ротация по `BACKUP_RETENTION_DAYS`. Настройка cron/systemd-
таймера и восстановление — `docs/BACKUP.md`.

## Тестирование

```bash
pip install -r requirements/dev.txt
pytest
```

31 тест покрывает ключевые сценарии сервисного слоя на изолированной
in-memory SQLite (без Docker и без сети):

- идемпотентность финализации платежа по webhook (повторный webhook не
  выдаёт номерки повторно; неверная подпись/неизвестный заказ/отказ — не
  приводят к выдаче);
- выдача номерков — уникальность и границы (`InsufficientTicketsError`),
  блокировка розыгрыша (`GiveawayLockedError`), накопление `tickets_issued`
  при нескольких выдачах подряд;
- аутентификация (успех/неверный пароль/заблокированный пользователь) и
  полная матрица прав ролей (Super Admin / Administrator / Operator);
- ручная (офлайн) регистрация — подтверждение с выдачей номерков, запрет
  повторного подтверждения, отмена только до подтверждения;
- выключение Telegram (`TELEGRAM_ENABLED=false`) — уведомления и рассылки не
  пытаются слать через выключенный канал, создать рассылку при выключенном
  Telegram нельзя.

## База данных

SQLAlchemy 2.x (async), Repository Pattern + сервисный слой. Модели —
`app/models/`.

Только SQLite — PostgreSQL и Alembic в проекте намеренно не поддерживаются
(осознанное упрощение архитектуры для одной небольшой инсталляции). Схема
создаётся автоматически при каждом старте backend'а —
`scripts/init_db.py` (`Base.metadata.create_all()`), без миграционного
инструмента.

Это значит:

- Новую таблицу create_all создаст сам, без дополнительных действий.
- Изменение существующей таблицы (добавить/удалить/переименовать колонку)
  create_all не применит — для сервера с реальными данными такое изменение
  нужно будет накатить вручную (разовый `ALTER TABLE`) или начать с чистой
  БД. Для проекта на одной SQLite-базе без Postgres это осознанный компромисс
  в пользу простоты.

```bash
# создать/проверить таблицы вручную (то же самое делает entrypoint backend'а)
python -m scripts.init_db
```

## Что проверено

Перед сдачей выполнена проверка (см. `docs/VERIFICATION.md` для деталей):

- Отсутствие `SyntaxError` / `ImportError` / `ModuleNotFoundError` во всём
  проекте (включая корректную кодировку файлов).
- Отсутствие циклических импортов — весь пакет `app/*`, `admin_api/*`,
  `bots/telegram/*` импортируется в чистом окружении без ошибок.
- Регистрация всех роутеров FastAPI (перечислены в `admin_api/main.py`)
  подтверждена интроспекцией `app.routes`.
- Корректность моделей SQLAlchemy и создание схемы через `create_all()` —
  проверено на чистой БД (`scripts/init_db.py`).
- Полный сквозной сценарий через HTTP (`httpx` + ASGITransport): логин →
  создание розыгрыша → открытие регистрации → попытка изменить неизменяемые
  параметры (ожидаемый `409`) → ручная регистрация → выдача номерков →
  dashboard → журнал аудита → идемпотентный webhook на неизвестный заказ.
- Обработчики Telegram-бота регистрируются без ошибок в изолированном
  окружении с реальным `aiogram`.
- `admin-frontend`: `tsc -b` (проверка типов) и `vite build` — без ошибок.
- Корректность `docker-compose.yml` (валидный YAML, все `COPY`-пути в
  Dockerfile'ах существуют в репозитории).

> В среде разработки не было доступа к Docker Engine, поэтому финальный шаг
> `docker compose up -d --build` не был физически выполнен — но каждый его
> отдельный шаг (установка зависимостей с нуля, создание схемы, импорты,
> сборка фронтенда) воспроизведён и проверен в изолированных окружениях.
> Рекомендуем выполнить `docker compose up -d --build` при первом
> развёртывании и свериться с `docs/VERIFICATION.md`.

## Известные ограничения и дальнейшее развитие

- **VK-интеграция** полностью удалена из проекта (не нужна в проде сейчас) —
  вернёмся к ней отдельно позже. Код можно восстановить из git-истории
  (модуль `bots/vk`, `app/integrations/vk_api.py`, поля `vk_user_id`/
  `vk_username` у участника, канал `VK`/`BOTH` у рассылок).
- **ВТБ-эквайринг** реализован по типовому паттерну (см. раздел выше) —
  сверить поля с реальным договором при подключении.
- **Очень крупные розыгрыши** (сотни тысяч номерков): текущий алгоритм
  случайной выдачи (`TicketService._pick_random_numbers`) вычисляет
  свободные номера как разность множеств в памяти. Для розыгрышей на
  сотни тысяч номерков — заменить на таблицу предварительно перемешанного
  пула номеров (интерфейс `issue_tickets` не изменится).
- Rate limiting на `/api/auth/login` реализован (см. "Production-hardening").
  Ротация/отзыв refresh-токенов — не реализованы (сейчас refresh-токен живёт
  до истечения TTL и не может быть отозван раньше времени), рекомендуется
  перед публичным продакшеном при повышенных требованиях к безопасности.
- **Только SQLite, без миграций**: изменение схемы БД на сервере с реальными
  данными потребует ручного `ALTER TABLE` (см. раздел «База данных») — это
  осознанный выбор в пользу простоты, а не забытая недоработка.
